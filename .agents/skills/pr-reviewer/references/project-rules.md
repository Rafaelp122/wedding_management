# Project Rules Reference

This document contains the strict architectural, testing, and business rules for the Wedding Management System. You MUST enforce these rules during your code review.

## 🏗️ Architectural Rules (Strict)

### Backend (Django Ninja)
1. **Service Layer Pattern (ADR-006):** Endpoints in `api.py` (routers) MUST ONLY call functions in `services.py`. **Business logic is strictly forbidden in the API layer.**
2. **Multi-tenancy (ADR-009):** Every service and query must explicitly filter by the company/tenant (e.g., `.for_tenant(user.company)` or filtering by planner/wedding). Cross-tenant data leakage is a critical security failure.
3. **Data Integrity (ADR-011):** Models inherit from `BaseModel` which executes `full_clean()` on `save()`. Do not bypass this. Use Decimal for money, never Float.
4. **Operations:** Always define `operation_id` in routers (e.g., `weddings_list`) for correct Orval generation.
5. **Typing:** Strict static typing is enforced. Type hints are mandatory.

### Frontend (React, Vite)
1. **API Consumption (ADR-012):** It is **FORBIDDEN** to use `fetch` or `axios` manually in components. You must use exclusively the hooks generated by Orval located in `src/api/generated/`.
2. **Feature-Based Architecture:** Code is organized by features in `src/features/<feature_name>/`.
3. **UI Components (shadcn/ui):** Base UI components reside in `src/components/ui/`. Prioritize composition and Tailwind utility classes rather than modifying the base `ui` folder files directly.
4. **Forms:** Use `react-hook-form` integrated with `zod` for validation.
5. **Icons:** Use exclusively `lucide-react`.

## 🧪 Testing Standards (Mandatory)

### Backend (Pytest)
1. **Factories Over DB:** It is **FORBIDDEN** to use `.objects.create()` in tests. Use the classes in `backend/apps/*/tests/factories.py`.
2. **Isolation:** Service tests must be unit/isolated. Every service function must have at least one success test and one failure/exception test.
3. **Multitenancy Check:** Always verify that User A cannot access or modify data belonging to User B's tenant.

### Frontend (Vitest & RTL)
1. **User-Centric:** Test user behavior. Prioritize accessibility queries (`getByRole`, `getByLabelText`).
2. **Mocking:** Mock Orval hooks using `vi.mock`. Never make real API calls in tests.
3. **Data:** Use `faker` for generating test payloads.

## 📐 Key Business Rules

1. **Zero Tolerance (BR-F01):** The sum of `Installments` must exactly equal the `Expense` total. The system adjusts the LAST installment to absorb cent differences.
2. **Legal Anchor (BR-F02):** An `Expense` linked to a `Contract` must have the exact same `actual_amount` as the contract's `total_amount`.
3. **Payment Consistency (BR-F03):** An installment marked as `PAID` must have a `paid_date`.
4. **Acquisition Decoupling (BR-L04):** The physical delivery status of an `Item` is independent of its financial payment status.
5. **Scheduler Automation (BR-S01):** `PAYMENT` events in the calendar are auto-generated from installments. Users cannot manually create or edit their dates directly in the calendar module.
