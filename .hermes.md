# Wedding Management — Hermes Rules

Hermes: este é o projeto Wedding Management. Siga estas regras ao trabalhar aqui.

## Comandos

- `make test` — rodar todos os testes (backend + frontend)
- `make lint` — verificar lint (ruff + eslint)
- `make check` — test + lint juntos
- `make build` — reconstruir containers
- `make format` — formatar código

## Regras de Código

### Backend (Django + Ninja)
- Sempre usar `services.py` — lógica de negócio nunca em `api.py`
- Multi-tenancy: `Model.objects.for_tenant(company)` em toda query
- Usar factories em `backend/apps/*/tests/factories.py` — nunca `.objects.create()`
- `mypy` deve passar antes de commit

### Frontend (React + TypeScript)
- Ler `docs/DESIGN.md` antes de trabalho de UI
- Usar Orval hooks de `src/api/generated/` — nunca `fetch`/`axios`
- shadcn/ui em `src/components/ui/` — componentes de negócio na feature
- `react-hook-form` + `zod` para formulários
- `lucide-react` para ícones

### Testes
- Backend: pytest com factories, testes unitários isolados
- Frontend: Vitest + RTL, importar de `@/test-utils`
- Todo `services.py` precisa de testes de sucesso e falha

## Workflow

- Commits: Conventional Commits (`feat(weddings): add endpoint`)
- ADRs: novo arquivo em `docs/ADR/` para mudanças estruturais
- Secrets: nunca commitar — `detect-secrets` ativo

## Skills do Projeto

Skills específicas estão em `.agents/skills/`. Carregar sob demanda:
- `wedding-backend` — padrões Django, services, models
- `wedding-frontend` — padrões React, componentes, hooks
- `wedding-business-rules` — regras de negócio
- `wedding-backend-testing` — padrões de teste backend
- `wedding-frontend-testing` — padrões de teste frontend

## Subagentes

Usar subagentes para mudanças em +2 arquivos, lógica multi-step, ou buscas complexas.
Conversa principal = apenas perguntas diretas, edits simples, coordenação.
